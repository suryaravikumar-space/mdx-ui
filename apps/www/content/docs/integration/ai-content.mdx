When an LLM generates MDX that includes mdx-ui component tags (`<Callout>`, `<Steps>`, `<DataTable>`, etc.), and you render that output with `next-mdx-remote` or contentlayer with the right component map registered, **the UI appears automatically** — no extra processing needed.

This page shows you how to wire the full pipeline.

---

## How it works

```
LLM API call
    │
    │  MDX string with component tags
    ▼
next-mdx-remote / compileMDX
    │
    │  looks up components={} map
    ▼
React renders <Callout>, <Steps>, <DataTable>…
    │
    ▼
Styled UI components appear on screen
```

The LLM generates text. Your renderer already knows about the components. Nothing extra needed in between.

---

## Step 1 — Register your components

After running `npx @ravikumarsurya/mdx-ui add callout steps datatable highlight mermaid`, register them in your component map:

```tsx
// components/mdx-components.tsx
import { Callout } from "@/components/mdx/callout"
import { Steps, Step } from "@/components/mdx/steps"
import { DataTable } from "@/components/mdx/data-table"
import { Highlight } from "@/components/mdx/highlight"
import { Mermaid, MermaidBST, MermaidTree, MermaidBFS, MermaidDFS } from "@/components/mdx/mermaid"
import { CodeBlock } from "@/components/mdx/code-block"
import { Tabs, TabsList, TabsTrigger, TabsContent } from "@/components/mdx/tabs"

export const mdxComponents = {
  Callout,
  Steps,
  Step,
  DataTable,
  Highlight,
  Mermaid,
  MermaidBST,
  MermaidTree,
  MermaidBFS,
  MermaidDFS,
  CodeBlock,
  Tabs,
  TabsList,
  TabsTrigger,
  TabsContent,
  pre: CodeBlock,
}
```

## Step 2 — Give the LLM the component catalog

Include this system prompt in your API call. The model will use our syntax instead of plain markdown.

```ts
const SYSTEM_PROMPT = `
You are a technical writer. Output MDX — Markdown with JSX components.
Use the following components instead of plain markdown equivalents:

CALLOUTS (instead of blockquotes):
<Callout type="info">Informational note</Callout>
<Callout type="warning">Caution message</Callout>
<Callout type="error">Error or danger</Callout>
<Callout type="success">Tip or success</Callout>

STEPS (instead of numbered lists for procedures):
<Steps>
  <Step>First action to take</Step>
  <Step>Second action to take</Step>
</Steps>

Steps can contain rich content including code blocks:
<Steps>
  <Step>
    **Clone the repository**
    \`\`\`bash
    git clone https://github.com/your-org/your-repo.git
    cd your-repo
    \`\`\`
  </Step>
  <Step>
    **Install dependencies**
    \`\`\`bash
    npm install
    \`\`\`
  </Step>
  <Step>
    **Start the development server**
    \`\`\`bash
    npm run dev
    \`\`\`
  </Step>
</Steps>

IMPORTANT: Never use numbered markdown lists (1. 2. 3.) when steps contain
code blocks — the markdown parser will break the list. Always use <Steps><Step>
for any procedure that includes commands or code examples.

TABLES (instead of markdown tables):
<DataTable
  headers={["Column A", "Column B", "Column C"]}
  rows={[
    ["value 1", "value 2", "value 3"],
    ["value 4", "value 5", "value 6"],
  ]}
/>

HIGHLIGHT (instead of bold for key terms):
<Highlight>key term</Highlight>
<Highlight color="blue">another term</Highlight>

TABS (for showing alternatives):
<Tabs>
  <TabsList>
    <TabsTrigger value="npm">npm</TabsTrigger>
    <TabsTrigger value="pnpm">pnpm</TabsTrigger>
  </TabsList>
  <TabsContent value="npm">\`\`\`bash
npm install example
\`\`\`</TabsContent>
  <TabsContent value="pnpm">\`\`\`bash
pnpm add example
\`\`\`</TabsContent>
</Tabs>

Always use fenced code blocks with language tags for code.
Output only the MDX content — no frontmatter, no explanation.
`
```

## Step 3 — Call the LLM and render the output

### With OpenAI

```ts
// app/api/generate/route.ts
import OpenAI from "openai"

const openai = new OpenAI()

export async function POST(req: Request) {
  const { topic } = await req.json()

  const completion = await openai.chat.completions.create({
    model: "gpt-4o",
    messages: [
      { role: "system", content: SYSTEM_PROMPT },
      { role: "user",   content: `Write a technical guide about: ${topic}` },
    ],
  })

  const mdx = completion.choices[0].message.content ?? ""
  return Response.json({ mdx })
}
```

### Render the MDX output

```tsx
// app/generate/page.tsx
import { MDXRemote } from "next-mdx-remote/rsc"
import { mdxComponents } from "@/components/mdx-components"

export default async function GeneratePage() {
  const res = await fetch("/api/generate", {
    method: "POST",
    body: JSON.stringify({ topic: "React Server Components" }),
  })
  const { mdx } = await res.json()

  return (
    <article className="prose max-w-3xl mx-auto py-10">
      <MDXRemote source={mdx} components={mdxComponents} />
    </article>
  )
}
```

When `MDXRemote` encounters `<Callout>`, `<Steps>`, or `<DataTable>` in the LLM output, it looks them up in `mdxComponents` and renders the full UI — automatically.

## Chatbot responses (fully dynamic)

Chatbot APIs (GPT, Claude, Gemini, etc.) return **plain markdown**, not MDX component tags. They also generate numbered lists with code blocks that are not correctly indented under their list items — which causes the CommonMark parser to break the list at the first code fence. Everything from item 2 onward renders as raw monospace text inside a broken code block.

### Why the standard parser breaks

```markdown
1. **Clone the repository**:
   ```bash
git clone https://github.com/your-org/app.git
```

2. **Install dependencies**:
   ```bash
npm install
```
```

The closing ` ``` ` at column 0 does not satisfy CommonMark's list continuation rules. The list terminates, items 2 and 3 are never parsed as list items, and the remark plugin's `transformSteps` never sees a complete `list` node to convert.

### The fix: `preprocessMarkdown`

`preprocessMarkdown` is a **string pre-processor** that runs before the MDX parser. It uses a line-by-line state machine to collect numbered items and their associated code blocks regardless of indentation, then emits them as `<Steps><Step>` MDX.

```bash
npm install @ravikumarsurya/remark-mdx-ui
```

```tsx
// app/chat/page.tsx
import { MDXRemote } from "next-mdx-remote/rsc"
import remarkMdxUi, { preprocessMarkdown } from "@ravikumarsurya/remark-mdx-ui"
import { mdxComponents } from "@/components/mdx-components"

interface Props {
  chatbotResponse: string // raw markdown from any chatbot API
}

export default function ChatResponse({ chatbotResponse }: Props) {
  // Step 1 — Fix broken numbered lists before the parser sees the string
  const fixed = preprocessMarkdown(chatbotResponse)

  // Step 2 — Let the remark plugin upgrade blockquotes, tables, ==highlights==
  return (
    <MDXRemote
      source={fixed}
      components={mdxComponents}
      options={{
        mdxOptions: {
          remarkPlugins: [
            [remarkMdxUi, { callout: true, table: true, highlight: true }],
          ],
        },
      }}
    />
  )
}
```

### What `preprocessMarkdown` handles

| AI chatbot generates | Rendered as |
| --- | --- |
| `1. **step** + code block (not indented)` | `<Steps><Step>` with code inside |
| `1. **step** + multiple code blocks` | single `<Step>` containing all blocks |
| `1. Step\n2. Step\n3. Step` (text only) | `<Steps><Step>` (no code needed) |
| Non-numbered content | passed through unchanged |

<Callout type="info">
  `preprocessMarkdown` only converts sequences of **2 or more** numbered items. A lone `1.` item is left as-is.
</Callout>

### Full chatbot pipeline

```
Chatbot API response (plain markdown)
    │
    │  preprocessMarkdown(response)
    ▼
Fixed MDX string — numbered lists → <Steps><Step>
    │
    │  MDXRemote + remarkMdxUi plugin
    ▼
Remark upgrades blockquotes → <Callout>, tables → <DataTable>
    │
    ▼
React renders all components with full UI
```

### Using with a streaming chatbot

```tsx
// app/api/chat/route.ts
import { preprocessMarkdown } from "@ravikumarsurya/remark-mdx-ui"

export async function POST(req: Request) {
  const { message } = await req.json()

  // Call your chatbot API (OpenAI, Anthropic, etc.)
  const response = await callChatbot(message)
  const rawMarkdown = response.content

  // Pre-process before sending to the client
  const processedMdx = preprocessMarkdown(rawMarkdown)

  return Response.json({ mdx: processedMdx })
}
```

```tsx
// app/chat/ChatMessage.tsx
"use client"
import { MDXRemote } from "next-mdx-remote"
import remarkMdxUi from "@ravikumarsurya/remark-mdx-ui"
import { mdxComponents } from "@/components/mdx-components"

export function ChatMessage({ mdx }: { mdx: string }) {
  return (
    <MDXRemote
      compiledSource={mdx}
      components={mdxComponents}
    />
  )
}
```

## Automatic upgrade: plain markdown → components

For **static files or controlled LLM output** (where you own the system prompt), the `@ravikumarsurya/remark-mdx-ui` remark plugin upgrades standard markdown patterns to components at AST level — no string preprocessing needed.

```tsx
import { MDXRemote } from "next-mdx-remote/rsc"
import remarkMdxUi from "@ravikumarsurya/remark-mdx-ui"
import { mdxComponents } from "@/components/mdx-components"

<MDXRemote
  source={aiGeneratedMarkdown}
  components={mdxComponents}
  options={{
    mdxOptions: {
      remarkPlugins: [
        [remarkMdxUi, { steps: true, highlight: true }]
      ],
    },
  }}
/>
```

What the plugin upgrades automatically:

| AI generates | Renders as |
| --- | --- |
| `> [!NOTE]` / `> [!WARNING]` | `<Callout type="info/warning">` |
| `> **Note:**` legacy syntax | `<Callout type="info">` |
| Markdown table `\| col \|` | `<DataTable headers rows />` |
| ` ```mermaid ` code block | `<Mermaid chart={…} />` (auto-detects type) |
| Numbered list (well-formed) | `<Steps><Step>` *(opt-in)* |
| `==text==` | `<Highlight>` *(opt-in)* |

<Callout type="warning">
  The remark plugin's `steps` transform only works on **well-formed** ordered lists — where code blocks are correctly indented under their list items. For chatbot output where code blocks escape the list, use `preprocessMarkdown` instead (see above).
</Callout>

## With contentlayer (static MDX files)

If you store AI-generated content as `.mdx` files and process them with contentlayer, add the plugin in `contentlayer.config.ts`:

```ts
import remarkMdxUi from "@ravikumarsurya/remark-mdx-ui"

export default makeSource({
  mdx: {
    remarkPlugins: [
      [remarkMdxUi, { steps: true }]
    ],
  },
})
```

## With any other LLM

The system prompt and rendering pipeline work with any model:

```ts
// Anthropic Claude
import Anthropic from "@anthropic-ai/sdk"
const client = new Anthropic()
const message = await client.messages.create({
  model: "claude-opus-4-5",
  system: SYSTEM_PROMPT,
  messages: [{ role: "user", content: topic }],
})
const mdx = message.content[0].type === "text" ? message.content[0].text : ""
```

```ts
// Google Gemini
import { GoogleGenerativeAI } from "@google/generative-ai"
const genAI = new GoogleGenerativeAI(process.env.GEMINI_API_KEY!)
const model = genAI.getGenerativeModel({ model: "gemini-1.5-pro",
  systemInstruction: SYSTEM_PROMPT })
const result = await model.generateContent(topic)
const mdx = result.response.text()
```
